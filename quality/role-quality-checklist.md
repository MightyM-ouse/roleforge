# Role Quality Checklist

Use this checklist for each role file.

## Required Structure

- [ ] Role Identity is present.
- [ ] Mission is present.
- [ ] Authority Level is present.
- [ ] Primary Responsibilities are present.
- [ ] Explicit Non-Responsibilities are present.
- [ ] Required Inputs are present.
- [ ] Valid Outputs are present.
- [ ] GitHub Permission is present.
- [ ] Refusal and Redirect Rules are present.
- [ ] Handoff Rules are present.
- [ ] Response Template is present.
- [ ] Good Request Examples are present.
- [ ] Bad Request Examples are present.
- [ ] Quality Checklist is present.
- [ ] Anti-Patterns are present.

## Boundary Quality

- [ ] The role has a clear purpose.
- [ ] Responsibilities do not overlap silently with another role.
- [ ] Non-responsibilities name likely misuse cases.
- [ ] Refusal behavior is clear and useful.
- [ ] Handoff behavior names likely target roles.

## GitHub Quality

- [ ] GitHub-ready output is separated from GitHub updates.
- [ ] GitHub update authority requires explicit user approval.
- [ ] Friendly Colleague, Project Mentor, and AI Tutor have no GitHub update authority.
- [ ] Formal roles only update GitHub inside their boundaries.

## Output Quality

- [ ] Required inputs are specific enough to prevent guessing.
- [ ] Valid outputs are role-specific.
- [ ] The response template is practical.
- [ ] Examples show both correct and incorrect usage.
- [ ] Anti-patterns cover the most likely role drift.
