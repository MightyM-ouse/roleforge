# RoleForge Review Checklist

Use this checklist to review a RoleForge repository or project adoption.

## Repository Completeness

- [ ] README explains RoleForge purpose and navigation.
- [ ] ROLEFORGE_MANIFEST.md explains the operating model.
- [ ] ROLE_INDEX.md provides quick role selection.
- [ ] All 11 role files exist.
- [ ] All 8 cross-role rule files exist.
- [ ] Role lifecycle guidance exists.
- [ ] All 7 response templates exist.
- [ ] Examples cover correct use, wrong use, handoffs, GitHub output, and refusal redirects.
- [ ] Structural validation guidance exists.
- [ ] Quality checklists exist.
- [ ] Project adoption guidance exists.

## Role Completeness

- [ ] Every role follows Role File Standard v1.
- [ ] Every role has required inputs and valid outputs.
- [ ] Every role has refusal and redirect rules.
- [ ] Every role has handoff rules.
- [ ] Every role has good and bad request examples.
- [ ] Every role has anti-patterns.
- [ ] Unauthorized role requests stop and redirect instead of partially completing.

## Permission Consistency

- [ ] Friendly Colleague has no GitHub update authority.
- [ ] Project Mentor has no GitHub update authority.
- [ ] AI Tutor has no GitHub update authority.
- [ ] Friendly Colleague, Project Mentor, and AI Tutor do not prepare GitHub-ready output.
- [ ] Formal roles require explicit approval before GitHub updates.
- [ ] GitHub-ready output is not treated as automatic posting permission.

## Adoption Safety

- [ ] Project-specific examples do not narrow RoleForge to one project.
- [ ] Project-specific rules do not weaken role boundaries.
- [ ] Magna Enso appears only as an adoption example.
- [ ] No implementation code or executable skills are introduced in this documentation layer.
